TASK 2 — Smart cache for hybrid signal
The hybrid signal pipeline currently re-runs the full 60/30/10 evaluation every cycle for every symbol. Most cycles produce identical results because indicator values barely move. A small cache eliminates redundant LLM escalations.
Create a new file backend/services/hybridCacheService.js with this surface:
	•	getCachedDecision(cacheKey) → { route, signal, cachedAt } | null
	•	setCachedDecision(cacheKey, route, signal) → void
	•	buildCacheKey({ symbol, regime, rsi, vwapDistance }) → string
	•	invalidateCacheForSymbol(symbol) → void (used after trade execution)
	•	getCacheStats() → { size, hitsToday, missesToday, hitRate }
Cache key format: {regime}|rsi{vwapBucket} where:
	•	rsiBucket = Math.floor(rsi / 10) * 10 (so 47.3 becomes 40, 52.1 becomes 50)
	•	vwapBucket = Math.round(vwapDistance * 200) / 200 (so 0.0073 becomes 0.005, 0.012 becomes 0.010)
	•	regime is the discrete label from regimeService (RISK_ON, RISK_OFF, NEUTRAL, etc.)
TTL: 120 minutes. Implementation: in-process Map with timestamp. Eviction on read when expires_at < now. Maximum 5,000 entries; LRU eviction on insert when at limit.
Cache scope: only cache the decideRoute() output AND the buildStatisticalSignal() / buildHoldSignal() output. Do NOT cache full Council deliberations — those are too expensive to risk staleness on.
Wire into hybridSignalService.js: at the top of the route-decision logic, compute the cache key and check getCachedDecision(). On hit, return the cached route + signal directly. On miss, run the full pipeline, then setCachedDecision() with the result before returning.
Cache invalidation hooks:
	•	On any trade execution for a symbol, call invalidateCacheForSymbol(symbol). Wire this into agent.js’s executeOrder() success path.
	•	On regime transition (regimeService publishes a different label than the previous cycle), invalidate the entire cache. Wire this in agent.js where regime is read each cycle.
	•	On any Discord-approved config change to confidence_gate_base, invalidate. Wire this into discordApprovalService.js after invalidateRuntimeConfig().
Stats: track hitsToday and missesToday (in-process counters, reset at daily reset in dailyReset()). Expose via a new endpoint GET /api/hybrid/cache-stats in server.js (gated by requireOperator).
Tests in tests/hybridCache.test.js:
	•	buildCacheKey produces stable keys for stable inputs (idempotent)
	•	buildCacheKey distinguishes inputs that should be distinct (different symbol, different RSI bucket, different VWAP bucket)
	•	get returns null on miss, returns cached value on hit
	•	TTL expiry: set, advance time (use vi.useFakeTimers), get returns null after 121 minutes
	•	LRU eviction: fill to 5,000, insert one more, oldest is evicted
	•	invalidateCacheForSymbol removes only that symbol’s entries
Documentation: in the file header, note that this cache is in-process only. A backend restart wipes it. This is acceptable because the upstream signals will quickly re-populate it within the first few cycles after restart.
TASK 3 — Regime meta-layer (raise-only)
Wire regimeService into dynamicGateService so that high-volatility or correlation-spike regimes can RAISE the confidence gate. The regime meta-layer can never lower the gate — only tighten safety.
Add a new function to dynamicGateService.js:
computeRegimeAdjustment(regimeData) → number (always >= 0)
regimeData is the object returned by regimeService.getCurrentRegime() (or equivalent — read regimeService to find the actual API). The adjustment rules:
	•	Default: 0 (no adjustment)
	•	If regime label is HIGH_VOLATILITY or VIX_SPIKE: +0.05
	•	If regime label is CORRELATION_SPIKE or RISK_OFF: +0.05
	•	If both conditions are true: +0.08 (capped, not additive to +0.10 which would push over the ceiling)
	•	If statistical confidence in regime classification is below 0.7: 0 (don’t trust ambiguous signals)
Important: this function returns the RAISE delta only. The actual gate computation is in getCurrentGate() and that’s where it gets composed with the base_gate, council_delta, and pin logic.
Modify getCurrentGate() in dynamicGateService.js:
The existing function returns clamp(base_gate + council_delta, [0.65, 0.90]) with pin override if active. The new behaviour:
effectiveGate = base_gate + council_delta + regimeAdjustment
clamped = clamp(effectiveGate, [0.65, 0.90])
finalGate = pinned ? max(clamped, pin_value) : clamped
Critical invariants — assert these in code with descriptive errors:
	•	regimeAdjustment >= 0 (never negative — would loosen the gate)
	•	finalGate >= 0.65 (defence-in-depth)
	•	finalGate <= 0.90 (defence-in-depth)
	•	If pinned, finalGate >= pin_value (pin can’t be undermined by regime drop)
Read the regime once per call. If regimeService throws or returns null, regimeAdjustment defaults to 0. Trading must never be blocked by a regime read failure. Log [REGIME-ADJUST] failure: <reason> on errors.
Logging: when regimeAdjustment changes from the previous cycle’s value, log [GATE-REGIME] adjustment 0.00 -> 0.05 (HIGH_VOLATILITY). Don’t log on every cycle; only on transitions, like the strategy overlay logging in Phase A.5.
Expose adjustment in getStatus() output: add regimeAdjustment, regimeLabel, regimeConfidence fields.
Tests in tests/regimeAdjustment.test.js:
	•	0 adjustment when regime is null/undefined
	•	0 adjustment when regime is NEUTRAL or RISK_ON
	•	+0.05 for HIGH_VOLATILITY
	•	+0.05 for RISK_OFF
	•	+0.08 for both HIGH_VOLATILITY and RISK_OFF
	•	0 when regime confidence < 0.7 regardless of label
	•	Effective gate respects clamp: base 0.85 + regime 0.08 = 0.93, clamps to 0.90
	•	Effective gate respects pin: pinned at 0.85, regime adds 0.05, returns 0.90 (pin floor + regime, clamped)
	•	Negative adjustments are rejected (assertion fires) — this should never happen but the test proves the invariant

TASK 6 — Asymmetric sizing by confidence
Currently position size is computed as a fixed percentage of equity (4% for day, 5% for swing). The new behaviour scales position size by Council confidence: higher confidence = larger size, within hard caps.
Formula: positionSize = 1.0 + 2.0 * ((confidence - 0.65) / 0.25), clamped to [1.0%, 3.0%]
Where confidence is the Judge’s final confidence from Council deliberation, expressed as a fraction (0.65 to 0.90). Examples:
	•	confidence 0.65 → size 1.0%
	•	confidence 0.75 → size 1.8%
	•	confidence 0.85 → size 2.6%
	•	confidence 0.90 → size 3.0% (clamped — the hard cap)
This applies ONLY to day strategies (day, asx_day). Swing strategies keep their existing literal sizing (5% for swing, 5% for asx_swing).
Important: this REPLACES the existing 4% per-name cap on day strategies in normal operation, but the 4% cap (now reconciled to 3% — see Task 8) still acts as a hard ceiling. Defence-in-depth: even if the formula returns a value outside [1%, 3%] due to a bug, the existing position-cap math in riskManager.js (qty = min(qtyByRisk, qtyByPosition)) still enforces the strategy’s maxPositionPct.
Implementation approach:
	1.	Create a new helper function in a new file backend/services/positionSizingService.js: computeAsymmetricSize(confidence, baseStrategy) → number (decimal percentage).
	•	If baseStrategy.name is not ‘day’ and not ‘asx_day’, return baseStrategy.maxPositionPct unchanged (swing pass-through).
	•	If confidence is null/undefined/NaN, return 0.01 (the floor — fail safe small).
	•	Apply the formula, clamp to [0.01, 0.03].
	2.	The existing flow is: agent.js builds sc via buildStrategyConfig(), passes sc into riskManager.evaluateBuy(). Modify agent.js so that for day/asx_day strategies, it overlays the asymmetric size onto sc.maxPositionPct BEFORE calling evaluateBuy. The overlay happens after Council deliberation when confidence is known, in the call site between Council and evaluateBuy.
	3.	Do not modify riskManager.js. The function continues to read sc.maxPositionPct as before — it just receives a different value depending on confidence.
	4.	Statistical-only signals (where _statisticalOnly: true and agreementCount: 1) should NOT receive asymmetric sizing — they fail quorum by design and never reach evaluateBuy with a BUY signal. This is preserved automatically since the overlay only applies to Council-driven BUYs. Verify by inspection.
	5.	Logging: when sizing differs from base, log [SIZING] AAPL day: confidence 0.82 → size 2.36% (base 0.04). Log per trade decision, not per cycle.
	6.	Defence-in-depth: in agent.js after computing the asymmetric size but before calling evaluateBuy, assert the result is within [0.005, 0.05] (broader than the formula range to allow for future rule changes). If out of range, log a critical error, fall back to base, and write an audit row event_type: ‘SIZING_BOUND_VIOLATED’.

TASK 7 — Per-symbol expectancy tracking with auto-suspend
Track rolling expectancy per (symbol, strategy) pair. Auto-suspend any pair whose expectancy is negative across the last 30 closed trades. Suspension means: that pair cannot enter new trades. Existing positions are unaffected — they exit normally.
New service: backend/services/expectancyService.js
Surface:
	•	recordTradeOutcome({ symbol, strategy, pnlPct }) → void (called from agent’s trade-close path)
	•	getExpectancy(symbol, strategy) → { trades: number, winRate: number, avgWin: number, avgLoss: number, expectancy: number, suspended: boolean, suspensionReason: string | null }
	•	isSuspended(symbol, strategy) → boolean (fast check, used in pre-trade hook)
	•	listSuspended() → Array<{ symbol, strategy, since, expectancy }>
	•	reinstate(symbol, strategy, operatorId) → void (manual reinstatement via Discord)
Storage: new table symbol_expectancy in db.js ensureSchema:
	•	id SERIAL PRIMARY KEY
	•	symbol TEXT NOT NULL
	•	strategy TEXT NOT NULL
	•	last_30_pnls JSONB NOT NULL DEFAULT ‘[]’ (array of decimal percentages, oldest first)
	•	expectancy NUMERIC
	•	suspended BOOLEAN DEFAULT FALSE
	•	suspended_at TIMESTAMPTZ
	•	suspended_reason TEXT
	•	updated_at TIMESTAMPTZ DEFAULT NOW()
	•	UNIQUE(symbol, strategy)
Index: (suspended) WHERE suspended = TRUE for fast listing.
Expectancy formula (for each (symbol, strategy) row):
expectancy = winRate * avgWin - lossRate * abs(avgLoss)
Where winRate is fraction of trades with pnlPct > 0 in last 30. avgWin is mean of positive pnlPct, avgLoss is mean of negative pnlPct (absolute value).
Auto-suspend trigger: in recordTradeOutcome, after appending the new pnl and trimming to last 30:
	•	If trades < 30: don’t evaluate suspension yet (insufficient data)
	•	If trades = 30 and expectancy < 0: set suspended = true, suspended_at = NOW(), suspended_reason = ‘negative_expectancy_30_trades’. Write audit row event_type: ‘SYMBOL_AUTO_SUSPENDED’ with payload including the 30 pnls.
	•	If suspended = true and operator manually reinstates via Discord: clear suspension flags, write SYMBOL_REINSTATED audit row, BUT keep tracking — the next 30 trades determine whether it auto-suspends again.
Wire into agent.js:
	1.	On every trade close (executeOrder for SELL, or stop-out, or end-of-day flatten), call expectancyService.recordTradeOutcome with the realised pnl.
	2.	Before entering a new BUY, call isSuspended(symbol, strategy). If suspended, skip the symbol with a TRADE_REJECTED audit row reason: ‘symbol_auto_suspended’. Do not proceed to Council.
	3.	The check should happen after the killSwitch and circuitBreaker checks but before the hybrid signal computation — saves cost.
Discord command: add a new command parser for Reinstate <SYMBOL> <strategy> in discordApprovalService.js. Auth: requires DISCORD_APPROVER_IDS membership. On success, calls expectancyService.reinstate() and writes the audit row.
Tests in tests/expectancy.test.js:
	•	recordTradeOutcome appends and trims to 30 (insert 35, verify only last 30 retained)
	•	expectancy formula: 60% wins of 1% each, 40% losses of -1% each → 0.6 * 1 - 0.4 * 1 = 0.2
	•	expectancy formula: 30% wins of 2% each, 70% losses of -1% each → 0.3 * 2 - 0.7 * 1 = -0.1
	•	suspended after 30 trades with negative expectancy
	•	not suspended at 29 trades even if expectancy is -0.5 (insufficient data)
	•	reinstate clears suspension flags
	•	after reinstate, the next 30 trades reset the evaluation window
