TASK 6 — Asymmetric sizing by confidence
Currently position size is a fixed percentage of equity (4% for day, 5% for swing). The new behaviour scales position size by Council confidence: higher Judge confidence equals larger position, within hard caps.
Formula: positionSize = 1.0 + 2.0 * ((confidence - 0.65) / 0.25), clamped to [1.0%, 3.0%]
Where confidence is the Judge’s final confidence from Council deliberation, expressed as a fraction (0.65 to 0.90). Worked examples:
	•	confidence 0.65 returns size 1.0%
	•	confidence 0.75 returns size 1.8%
	•	confidence 0.85 returns size 2.6%
	•	confidence 0.90 returns size 3.0% (clamped at the ceiling)
This applies ONLY to day strategies (day, asx_day). Swing strategies (swing, asx_swing) keep their existing literal sizing of 5%.
Important: the existing per-name cap on day strategies (currently 4% in strategies.js, being reconciled in Task 9) acts as a hard ceiling outside this formula. Defence-in-depth: even if the formula returns a value outside [1%, 3%] due to a bug, the existing position-cap math in riskManager.js (qty = min(qtyByRisk, qtyByPosition)) still enforces the strategy’s maxPositionPct. We are layering, not replacing.

	•	If confidence is null, undefined, or NaN, return 0.01. Fail safe small.
	•	Apply the formula. Clamp to [0.01, 0.03]. Return.
	2.	Wire into agent.js. The existing flow is: agent.js builds sc via buildStrategyConfig() → passes sc into riskManager.evaluateBuy(). The clone semantics from Phase A.5 apply — never mutate the base STRATEGIES literal, never mutate a sc that’s shared across strategies.

TASK 7 — Per-symbol expectancy tracking with auto-suspend
Track rolling expectancy per (symbol, strategy) pair. Auto-suspend any pair whose expectancy is negative across the last 30 closed trades. Suspension means: that pair cannot enter NEW trades. Existing positions are unaffected — they exit normally via stops, trailing, or end-of-day flatten.
Create a new service backend/services/expectancyService.js with this surface:
	•	recordTradeOutcome({ symbol, strategy, pnlPct }) returns void. Called from agent’s trade-close path.
	•	getExpectancy(symbol, strategy) returns { trades, winRate, avgWin, avgLoss, expectancy, suspended, suspensionReason }
	•	isSuspended(symbol, strategy) returns boolean. Fast check, used in pre-trade hook.
	•	listSuspended() returns Array of { symbol, strategy, since, expectancy }
	•	reinstate(symbol, strategy, operatorId) returns void. Manual reinstatement via Discord.
New table symbol_expectancy in db.js ensureSchema:
	•	id SERIAL PRIMARY KEY
	•	symbol TEXT NOT NULL
	•	strategy TEXT NOT NULL
	•	last_30_pnls JSONB NOT NULL DEFAULT ‘[]’ (array of decimal percentages, oldest first)
	•	expectancy NUMERIC
	•	suspended BOOLEAN DEFAULT FALSE
	•	suspended_at TIMESTAMPTZ
	•	suspended_reason TEXT
	•	updated_at TIMESTAMPTZ DEFAULT NOW()
	•	UNIQUE(symbol, strategy)
Index: CREATE INDEX idx_symbol_expectancy_suspended ON symbol_expectancy(suspended) WHERE suspended = TRUE — fast listing of suspended pairs without scanning the full table.
Expectancy formula (computed inside recordTradeOutcome after each insert):
	•	winRate = count(pnls > 0) / total
	•	lossRate = count(pnls <= 0) / total
	•	avgWin = mean(pnls where pnl > 0), or 0 if no wins
	•	avgLoss = mean(abs(pnls) where pnl <= 0), or 0 if no losses
	•	expectancy = (winRate * avgWin) - (lossRate * avgLoss)
Auto-suspend trigger logic, inside recordTradeOutcome AFTER appending the new pnl and trimming to last 30:
	•	If trades < 30: do not evaluate suspension. Insufficient data.
	•	If trades = 30 and expectancy < 0 and not currently suspended: set suspended=true, suspended_at=NOW(), suspended_reason=‘negative_expectancy_30_trades’. Write audit row event_type ‘SYMBOL_AUTO_SUSPENDED’ with payload including the 30 pnls and the computed expectancy. Post a Discord notification via discordService.
	•	If currently suspended: do nothing — operator must reinstate manually. Continue tracking pnls for future evaluation post-reinstatement.
Reinstate behaviour:
	•	Clears suspended, suspended_at, suspended_reason. Writes audit row event_type ‘SYMBOL_REINSTATED’ with payload including the operator ID.
	•	Does NOT clear last_30_pnls. The next 30 trades determine whether it auto-suspends again. If reinstated when at 30 trades and still negative, the very next trade will re-evaluate — meaning it will auto-suspend again immediately after one more trade. This is intentional: the operator’s reinstate is a “give it one more chance” signal, not a permanent bypass.
Wire into agent.js:
	1.	On every trade close (executeOrder for SELL, stop-out path, end-of-day flatten path), call expectancyService.recordTradeOutcome({ symbol, strategy, pnlPct }) with the realised pnl. There may be multiple close paths — ensure all of them call this. The pnlPct is decimal (e.g. 0.012 for +1.2%).
	2.	Before entering a new BUY, call isSuspended(symbol, strategy). If suspended, skip the symbol with a TRADE_REJECTED audit row payload.reason=‘symbol_auto_suspended’. Do not proceed to hybrid signal computation. The check should happen AFTER the killSwitch and circuitBreaker checks but BEFORE the hybrid signal computation. This saves cost.
	3.	Discord command: add a new command parser for “Reinstate <SYMBOL> <strategy>” (case-insensitive) in discordApprovalService.js or discordChatService.js (whichever owns the command parsing). Auth: requires DISCORD_APPROVER_IDS membership, fail-closed semantics preserved. On success, calls expectancyService.reinstate() and replies with confirmation. On unauthorised, writes audit row ‘REINSTATE_UNAUTHORIZED’.
