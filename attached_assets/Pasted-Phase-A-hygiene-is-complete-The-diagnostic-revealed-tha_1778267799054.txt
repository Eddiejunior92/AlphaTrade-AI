TASK 5 — Per-cycle strategy overlay for max_holdings_day and max_position_pct
This is the most surgically delicate task. Read carefully.
The strategy config object sc comes from strategies.js as a literal. It is passed by reference into riskManager.js functions. Mutating sc in place would create cross-cycle leakage. Instead, agent.js must construct a per-cycle copy with overlays applied.
In agent.js, in runCycle, find the line where sc is selected for the day strategy (look for ‘day’ strategy logic — likely a sc = STRATEGIES.day or similar reference). For both day and asx_day strategies:
	1.	Replace the direct strategy reference with a new helper buildStrategyConfig(strategyName) defined in agent.js (not in strategies.js — this is runtime composition).
	2.	buildStrategyConfig returns a shallow clone of the base strategy object with overlays:
	•	If strategyName is ‘day’ or ‘asx_day’:
	•	rt = getRuntimeConfig()
	•	clone = { …STRATEGIES[strategyName] }
	•	if rt.max_holdings_day is non-null and within [1, 20], clone.maxHoldings = rt.max_holdings_day
	•	if rt.max_position_pct is non-null and within [0.005, 0.05] (the SAFE_KEYS hard cap), clone.maxPositionPct = rt.max_position_pct
	•	return clone
	•	For ‘swing’ and ‘asx_swing’: return base strategy unchanged (these aren’t in SAFE_KEYS yet)
	3.	Pass the cloned object into all downstream calls (evaluateBuy, evaluateSell, etc.). Do not mutate STRATEGIES[strategyName] itself.
	4.	Critical: log every overlay application once per change. Track the previously-applied overlay values in agent.js module state. When buildStrategyConfig produces values that differ from the last-logged values, emit [STRATEGY-OVERLAY] day: maxHoldings 6 -> 8, maxPositionPct 0.04 -> 0.045. This makes runtime changes visible in deploy logs without log spam.
	5.	Bounds-check defensively. If runtimeConfig returns out-of-range values (corrupted DB row, env-var typo), log a warning and use the base strategy value. Never let a bad config widen risk.
